## 1. 项目概述

Hermes 是公司已有的 AI Agent 基础系统。本项目目标是在现有 Hermes agent 与已配置向量数据库的基础上，将 Hermes 升级为企业“内核型 AI Agent”。

升级后的 Hermes 不应只是普通聊天机器人，而应成为公司专属 AI 中枢，具备长期记忆、知识检索、结构化事实查询、权限治理、可引用回答、增量更新、版本管理、审计追踪和质量评测能力。

长期来看，Hermes 的目标不是停留在“文档检索器”，而是逐步演进为嵌入企业运行结构的全功能企业级 Agent。该 Agent 应能理解多模态企业资料，包括 Word、PDF、Excel、PowerPoint、图片、扫描件、会议录音和后续企业系统数据；能把会议、文档、项目、合同、人员、客户和任务沉淀为可追溯的长期记忆；并在人工确认边界内支持经营分析、项目跟进、投标审查、会议决策追踪和企业工作流辅助。

系统建设重点不是扩大大模型上下文窗口，也不是单纯堆叠向量数据库，而是建设一套企业长期记忆系统。

企业长期记忆系统至少应包含：

1. 文档知识层。
2. 结构化事实层。
3. 会话记忆层。
4. 检索层。
5. 权限治理层。

本项目采用分阶段落地策略。当前阶段优先建设稳定的记忆底座和文档问答闭环；Excel、PowerPoint、音频会议理解、跨系统工作流和更强企业级决策辅助应作为后续阶段逐步纳入，不应在记忆底座尚未稳定前一次性展开。

## 2. 术语定义

### 2.1 Hermes

公司已有 AI Agent，作为本项目的 Agent 交互与编排基础。

### 2.2 企业长期记忆系统

用于长期存储、检索、治理和追溯企业知识的系统。它不是单一向量数据库，而是由文档知识、结构化事实、会话记忆、检索能力、权限治理、版本管理和审计能力共同构成。

### 2.3 文档知识层

保存企业非结构化和半结构化知识，包括公司文件、招标资料、公众号文章、制度文件、方案文档、合同模板、项目资料等。

### 2.4 结构化事实层

保存可精确查询的事实数据，例如客户、项目、合同、资质、招标要求、人员、日期、金额、状态等。

### 2.5 会话记忆层

保存用户与 Hermes 的交互上下文、长期偏好、任务状态、历史确认事实和项目上下文。

### 2.6 Hybrid Search

混合检索，指结合 Dense Retrieval、BM25/Sparse Retrieval、metadata filter、rerank、query rewrite 等能力的检索方式。

### 2.7 Citation

回答引用，指 Hermes 在回答中明确展示依据来源，包括文件名、版本、页码、章节、段落、更新时间等。

### 2.8 Rerank

重排，指对初步召回的候选内容进行相关性排序，提升最终进入上下文的内容质量。

### 2.9 Metadata Filter

基于文档类型、项目、客户、时间、版本、权限、密级等元数据进行过滤。

### 2.10 Versioning

版本管理，指对文档、chunk、事实、索引的版本进行识别、记录、失效和追溯。

### 2.11 多模态企业资料摄取

多模态企业资料摄取指对办公文档、表格、演示文稿、图片、扫描件、音频会议和后续视频资料进行解析、结构化、索引、引用和权限治理。该能力不等同于临时让模型读取附件，而是要求解析结果能进入长期记忆、可检索、可追溯、可评测。

### 2.12 会议记忆

会议记忆指对会议录音、转写文本、发言人、议题、决策、待办、风险和后续跟进事项进行结构化沉淀。会议记忆应支持后续查询“某次会议决定了什么”“谁负责什么”“哪些方向需要继续推进”，并与项目、客户、合同和文档知识关联。

### 2.13 BIM 数据管家

BIM 数据管家指对公司 BIM 建模文件和派生资料进行资产目录、版本、权限、项目归属、检索、引用和审计治理。该能力的初期目标不是解析几个 TB 的原始模型内容，也不是替代 BIM 协同平台或文件服务器，而是让 Hermes 能管理模型文件资产、记录可追溯元数据、索引派生文本 / 清单 / 说明，并为后续 IFC、RVT、NWD、DWG 等模型内容解析预留入口。

## 3. 项目背景

公司未来需要将 Hermes 接入公司全部文件、招标资料、公司公众号全部文章，并逐步扩展到更多企业系统。

当前已知情况：

1. 已有 Hermes agent 可作为基础。
2. 已配置向量数据库。
3. 当前向量维度为 1024。
4. 当前目标是打造企业级长期记忆系统，而不是只依赖上下文窗口。
5. 当前更关注“检索不到、知识遗忘、回答无依据、版本混乱和权限失控”问题，而不是数据库容量增长问题。
6. 公司存在 TB 级 BIM 建模文件资产，初期需要先建立目录治理、版本追踪、权限和项目归属，而不是一次性把全部模型内容解析进长期记忆。

企业知识具有以下特点：

1. 来源复杂。
2. 格式多样。
3. 权限敏感。
4. 版本频繁变化。
5. 需要可追溯回答。
6. 查询中包含大量中文专有名词、客户名、项目名、编号、金额、日期和条款。
7. 需要长期沉淀，而不能依赖短期上下文。

因此，本项目需要建设企业知识底座、检索增强系统、权限治理体系和可引用问答能力。

## 4. 项目目标

### 4.1 总体目标

将 Hermes 升级为企业内核型 AI Agent，使其成为公司知识、记忆、检索和智能问答的统一入口。

### 4.2 业务目标

1. 支持员工快速查询公司文档、招标资料、公众号文章和项目知识。
2. 支持投标、方案、项目、市场、管理等高频业务场景。
3. 降低知识查找成本。
4. 降低企业知识遗忘风险。
5. 提高知识复用效率。
6. 提高回答可信度和可追溯性。
7. 支持权限合规和审计追踪。

### 4.3 技术目标

1. 建立文档知识层。
2. 建立结构化事实层。
3. 建立会话记忆层。
4. 建立 hybrid search 检索层。
5. 建立 metadata filter、rerank、citation、versioning、permission control 和 evaluation 能力。
6. 支持增量更新。
7. 支持未来接入更多企业系统。

## 5. 项目范围

本项目范围包括：

1. 文档接入。
2. 文档解析。
3. 文档清洗。
4. 结构化 chunking。
5. 向量索引。
6. BM25 全文索引。
7. Hybrid search。
8. Metadata filter。
9. Rerank。
10. Query rewrite。
11. 可引用问答。
12. 结构化事实查询。
13. 文档版本管理。
14. 增量更新。
15. 权限控制。
16. 审计日志。
17. 评测系统。
18. 基础运维监控。
19. BIM 建模文件资产目录治理与后续模型解析入口规划。

## 6. 非本期范围

以下内容不纳入 MVP：

1. 全公司所有业务系统一次性接入。
2. OA、ERP、CRM 深度双向操作。
3. 自动替代人工审批。
4. 自动生成并直接提交投标最终文件。
5. 大规模知识图谱自动构建。
6. 私有大模型从零训练。
7. 完整多 Agent 自治平台。
8. 跨集团商业化多租户平台。
9. 财务、法务、投标结果的自动最终决策。
10. 完整音频会议理解与会议记忆闭环。
11. 完整 Excel / PowerPoint 结构化解析和跨表格推理。
12. 面向经营决策的全自动战略判断和公司发展方向自动决策。
13. 全量 TB 级 BIM 原始模型内容解析。
14. BIM 在线查看器、碰撞检测、自动算量、自动设计审查。
15. 替代 BIM 协同平台、文件服务器、NAS 或专业建模软件。

## 7. 用户角色

### 7.1 公司管理层

查询公司知识资产、项目进展、投标情况、经营材料和历史决策依据。

### 7.2 投标 / 商务人员

查询招标文件、资质要求、评分标准、历史投标资料、响应模板和报价参考。

### 7.3 项目经理

查询项目资料、交付记录、客户沟通、历史方案、风险事项和项目经验。

### 7.4 技术 / 方案人员

查询技术方案、产品能力、实施经验、历史案例、标准文档和技术响应材料。

### 7.5 市场 / 公众号运营人员

查询公司公众号文章、品牌素材、案例表述、对外宣传内容和历史发布记录。

### 7.6 法务 / 合规人员

查询合同条款、制度文件、审批依据、权限访问日志和敏感内容访问记录。

### 7.7 知识管理员

负责文档接入、知识质量管理、版本维护、权限配置、评测集维护和数据治理。

### 7.8 系统管理员

负责系统部署、数据源连接、权限集成、日志监控、任务调度和故障处理。

## 8. 核心业务场景

1. 查询某个招标项目的资质、人员、业绩、时间节点和评分标准。
2. 根据招标文件总结投标风险点，并引用原文。
3. 查询公司是否具备某类项目资质或历史案例。
4. 查找类似项目的投标方案、技术响应和商务响应。
5. 对比同一文件不同版本的变化。
6. 查询公众号是否发布过某主题文章，并提取可复用表述。
7. 根据客户名称查询相关项目、历史沟通、合同和交付资料。
8. 针对技术问题检索内部文档和历史方案，并生成带引用答案。
9. 查询某项制度的最新有效版本。
10. 基于长期记忆回答“公司之前如何处理过类似问题”。
11. 权限不足用户查询敏感资料时，系统拒绝或仅返回可见范围。
12. 知识管理员上传新文档后，系统自动解析、分块、索引并进入可检索状态。
13. 对低置信度回答进行标记，并进入人工复核或评测闭环。
14. 管理层查询某类项目经验，系统联合文档和结构化事实回答。
15. 审计人员追踪某次回答使用了哪些文档、版本和检索结果。
16. 会议结束后，系统可基于录音或转写材料生成会议摘要、决策点、行动项、风险点和后续跟进建议，并将可信内容写入长期记忆。
17. 投标或项目团队上传 Excel 报价表、评分表、项目台账和 PowerPoint 方案材料后，系统能保留表格结构、页码、标题和引用位置，并支持后续检索和问答。
18. 管理层基于文档知识、会议记忆和结构化事实，查询公司阶段性经营风险、项目推进状态和未来发展方向建议；系统必须明确区分文档依据、会议依据、结构化事实和模型归纳。
19. 项目团队查询某项目、楼栋、楼层、专业或责任人关联的 BIM 建模文件，系统返回模型文件资产、版本、路径 / 存储位置、权限标签和派生元数据引用。

## 9. 功能需求

### 9.1 数据接入

系统应支持本地文件目录、企业网盘、PDF、Word、Excel、PowerPoint、Markdown、HTML、TXT、公司公众号文章、招标资料压缩包、BIM 建模文件资产目录和后续企业系统 API。

后续阶段应支持音频会议资料接入，包括会议录音上传、转写、发言人区分、议题切分、决策和行动项抽取。若底层模型具备音频理解能力，可作为转写或理解 provider 之一，但企业级落地仍应保存可审计的转写文本、结构化会议记录和引用来源。

每条接入数据应记录数据源类型、原始路径或 URL、上传人、所属部门、所属项目、权限标签、文件 hash、接入时间、最近更新时间和文档版本。

BIM 文件资产接入应优先记录项目、楼栋、楼层、专业、模型类型、版本、原始路径 / 存储位置、文件大小、文件 hash、创建 / 修改时间、责任人、权限标签和派生元数据来源。原始模型文件不应直接进入 LLM 上下文；Hermes 初期只管理目录元数据、派生文本、结构化属性和可追溯链接。

### 9.2 文档解析

系统应支持 PDF 文本解析、OCR 兜底解析、Word 标题/段落/表格解析、Excel sheet 和行列结构解析、PowerPoint 页面/标题/正文/备注解析、HTML / 公众号正文提取、压缩包递归解析、图片和扫描件 OCR、表格结构保留和章节层级保留。

Excel 解析应尽量保留 sheet 名、表头、行列坐标、合并单元格、公式、金额和单位。PowerPoint 解析应尽量保留页码、页面标题、文本框、备注、图表说明和图片 OCR。会议音频解析应优先保留转写文本、时间戳、发言人、议题、决策点和行动项。

BIM 建模文件初期不要求完整解析模型几何、构件和族库内容；仅要求保留文件级元数据、派生清单、说明文档、导出表格或后续解析 provider 生成的结构化属性，并在回答中明确引用模型文件、版本、路径 / 存储位置和派生元数据来源。

### 9.3 结构化 Chunking

Chunking 应结合中文企业文档结构，不能简单按固定 token 切割。

系统应支持按标题层级、招标章节、制度条款、表格块、公众号小标题切分，保留父级标题路径、页码、章节号、段落号、chunk 与原文位置映射、适度上下文 overlap，并对长表格和长条款进行特殊处理。

每个 chunk 至少应包含 chunk_id、document_id、version_id、text、title_path、page_start、page_end、section_path、source_uri、permission_tags、created_at、updated_at、content_hash、embedding_id 和 sparse_index_id。

### 9.4 知识检索

系统应支持自然语言语义检索、关键词精确检索、项目名/客户名/编号/金额/日期检索、文档类型过滤、时间范围过滤、权限过滤、最新版本过滤、历史版本查询、多轮上下文检索、查询改写、查询扩展和结果重排。

### 9.5 Hybrid Search

系统必须支持 hybrid search。纯向量检索不能作为完整检索方案，原因包括：

1. 企业查询包含大量编号、项目名、客户名、金额、日期和专有名词。
2. 中文短查询容易语义漂移。
3. 招标条款和制度条款需要原文精确定位。
4. 向量检索不能解决权限过滤。
5. 向量检索不能解决版本过滤。
6. 向量检索不能替代结构化事实查询。
7. 向量检索不能保证引用精度。

Hybrid search 应包含 Dense Retrieval、BM25 / Sparse Retrieval、Metadata Filter、Rerank、Query Rewrite、规则召回和结构化事实召回。

### 9.6 Rerank

Rerank 应用于初步召回后，对候选 chunk 进行相关性重排。系统应支持中文 reranker、query 与 chunk 相关性评分、TopN 控制、版本权重、文档可信度权重、权限过滤后的重排和失败降级。

### 9.7 问答与引用

Hermes 回答应基于检索结果，展示引用来源。引用应包含文件名、版本、页码、章节、发布时间或更新时间。系统应在无法确认时明确说明不确定，在多来源冲突时提示冲突，对过期文档标记历史版本，对权限不足内容不得泄露，并支持“仅基于引用回答”模式和置信度提示。

### 9.8 结构化事实查询

系统应建设结构化事实层，支持实体和关系查询。

典型实体包括公司、客户、项目、招标项目、合同、产品、方案、资质、人员、部门、案例、公众号文章、制度文件和 BIM 模型文件资产。

典型关系包括客户拥有项目、项目关联合同、项目使用方案、招标文件要求资质、公司具备资质、人员参与项目、文档描述实体、版本替代版本和文章引用案例。

### 9.9 权限控制

权限控制必须贯穿数据接入、文档存储、chunk 存储、元数据查询、检索召回、prompt 构建、回答生成、引用展示、会话记忆和审计日志。

权限模型采用 RBAC + ABAC。RBAC 管理角色和基础权限，ABAC 基于部门、项目、密级、文档类型、用户属性进行细粒度控制。

### 9.10 版本管理

系统应支持文档版本识别、同名文件版本合并、hash 变更检测、最新有效版本标记、历史版本保留、版本差异对比、废止状态、生效时间、失效时间和用户指定历史版本查询。

### 9.11 增量更新

系统应支持文件新增检测、文件修改检测、文件删除或归档检测、公众号文章增量抓取、增量解析、增量 chunk 更新、增量 embedding、增量 BM25 索引更新、旧 chunk 失效标记和失败重试。

### 9.12 审计日志

系统应记录用户查询、检索命中结果、实际进入 prompt 的 chunk、回答引用来源、权限过滤结果、被拒绝访问的资源、文档上传/修改/删除、索引构建任务、管理员配置变更、模型调用信息、错误和异常。

### 9.13 评测系统

评测系统应覆盖 Recall@K、Precision@K、MRR、NDCG、Rerank 效果、引用准确率、回答忠实度、幻觉率、权限泄露测试、版本正确性测试、增量更新正确性和中文问法鲁棒性。

## 10. 非功能需求

### 10.1 性能

1. 普通知识问答 P95 小于 8 秒。
2. 检索链路 P95 小于 2 秒。
3. Rerank P95 小于 3 秒。
4. 普通文档上传后 5 分钟内可检索。
5. 支持百万级 chunk 扩展。

### 10.2 稳定性

系统应支持文档摄取任务可重试、索引写入幂等、单文档解析失败不影响批量任务、模型服务失败可降级、向量检索失败可降级到 BM25、Rerank 失败可使用初排结果。

### 10.3 可扩展性

系统应支持数据源插件化、解析器插件化、embedding 模型可替换、reranker 可替换、向量数据库可替换、权限策略可扩展和 Agent tool 可扩展。

### 10.4 安全性

系统应支持用户认证、统一身份接入、权限校验、数据加密传输、敏感字段脱敏、操作审计、prompt 注入防护、文档内容越权防护和管理后台权限隔离。

### 10.5 可观测性

系统应支持查询日志、检索日志、引用日志、模型调用日志、任务执行日志、错误日志、权限拒绝日志、延迟监控、命中率监控和评测指标看板。

## 11. 数据范围规划

### 11.1 Phase 1 数据范围

公司核心制度文件、近 1-2 年招标资料、典型项目方案、公司公众号历史文章、常用资质、案例、产品资料。

### 11.2 Phase 2 数据范围

历史投标文件、合同模板、项目交付文档、客户资料、会议纪要、市场素材、内部培训资料。

### 11.3 Phase 3 数据范围

OA、CRM、ERP、项目管理系统、邮件、即时通讯记录、财务和经营数据、外部行业知识源、BIM 建模文件资产目录与派生元数据。

### 11.4 BIM 数据管家后置专项范围

BIM 数据管家作为后置专项规划线，初期只纳入文件级资产目录和派生索引，不搬迁原始模型文件，不全量解析 TB 级模型内容。建议优先选择少量项目目录做试点，验证项目 / 楼栋 / 楼层 / 专业 / 版本 / 责任人 / 权限标签 / 文件 hash / 存储位置等元数据治理，再逐步评估 IFC、RVT、NWD、DWG 等格式的解析 provider。

## 12. MVP 范围

MVP 目标是形成可验证的企业知识问答闭环。

MVP 包含招标资料接入、公司核心文档接入、公众号文章接入、文档解析、结构化 chunking、metadata 生成、1024 维 embedding 写入现有向量数据库、BM25 全文索引、hybrid search、metadata filter、rerank、可引用回答、基础权限控制、文档版本记录、增量更新、审计日志和最少 100 条评测问题。

MVP 不包含全量企业系统接入、深度业务系统写操作、完整知识图谱、自动审批、自动投标文件最终提交和 TB 级 BIM 模型文件治理全量落地。

## 13. 验收标准

1. 至少完成内部文档、招标资料、公众号文章 3 类数据源接入。
2. 支持 PDF、Word、Excel、PowerPoint、HTML 基础解析。
3. 支持结构化 chunking 和元数据提取。
4. 支持 dense + BM25 hybrid search。
5. 支持 metadata filter。
6. 支持 rerank。
7. 支持回答引用来源。
8. 支持文档版本管理。
9. 支持增量更新。
10. 支持基础权限控制。
11. 支持审计日志。
12. 建立不少于 100 条高质量评测问题。
13. 核心场景 Top5 检索命中率达到 80% 以上。
14. 引用准确率达到 85% 以上。
15. 权限测试不得出现越权召回。
16. 用户可在 Hermes 中完成真实业务问答闭环。

## 14. 风险与边界

### 14.1 主要风险

1. 文档质量差，扫描件、图片和复杂表格解析难度高。
2. 文件版本命名混乱。
3. 早期元数据缺失。
4. 纯向量检索导致精确问题漏召回。
5. 权限模型不完善导致敏感信息泄露。
6. 评测集不足导致优化不可衡量。
7. 旧索引未失效导致回答使用过期知识。
8. 结构化事实抽取错误导致错误归因。
9. 中文企业专有名词和简称标准化困难。

### 14.2 边界说明

1. Hermes 是企业 AI 知识中枢，不是 OA、CRM、ERP 的替代品。
2. 向量数据库只是长期记忆系统的一部分，不是完整记忆系统。
3. 关键法律、财务、投标最终决策必须保留人工审批。
4. 对低置信度、来源冲突、权限不足的问题，系统必须明确提示。
