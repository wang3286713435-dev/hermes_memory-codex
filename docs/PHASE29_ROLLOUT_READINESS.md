# Phase 2.9 局部默认启用运行期基线维持与更高一级 Rollout 评估准备

## 1. 阶段目标

本阶段目标不是扩大当前启用范围，也不是直接进入更高一级 rollout 实施。

本阶段只回答以下问题：

1. 当前局部默认启用运行基线，是否已经积累了足够稳定的运行期证据
2. 这些证据是否足以作为更高一级 rollout 评估的输入
3. 当前还存在哪些风险，会阻止进入更高一级 rollout 评估

## 2. 当前运行基线摘要

当前运行基线保持不变：

- `RERANK_PROVIDER=aliyun`
- `ALIYUN_RERANK_MODEL=gte-rerank-v2`
- `ALIYUN_RERANK_TIMEOUT_MS=600`
- `RERANK_ENABLED=true`
- `RERANK_DEFAULT_ENABLEMENT_ENABLED=true`

当前启用条件保持不变：

1. request 侧 `source_type / route_type` 命中目标范围
2. query 命中关键词集合
3. candidate count 达到阈值

当前启用范围保持不变：

1. 招标资料相关 query
2. 资质要求类 query
3. 条款定位类 query
4. 时间安排 / 商务条款 / 评分标准类高歧义招标 query

当前不启用范围保持不变：

1. 非招标资料 query
2. 未命中关键词的普通事实型 query
3. candidate 不足的 query
4. 全局关闭 / 局部关闭场景

## 3. 当前运行观测摘要

### 3.1 规则层

截至 Phase 2.8 收口，规则层已经满足以下结论：

1. 命中集在正常窗口稳定不漂移
2. 误命中已清零
3. 未新增漏命中边界样本
4. request 侧 `source_type / route_type + 关键词` 的收紧规则已稳定生效

### 3.2 运行层

截至 Phase 2.8 收口，运行层已经积累以下证据：

1. `rerank_hit_rate` 保持稳定
2. `rerank_skip_rate` 保持稳定
3. 正常窗口下 `fail_open_rate` 持续接近 `0`
4. `global_disabled`、`local_disabled`、`timeout_drill` 均稳定可用
5. trace 可以区分：
   - 是否命中策略
   - 命中原因
   - 是否真正执行 rerank
   - 是否发生 `failed_open`
   - `provider / model / error_type / policy_reason`

### 3.3 延迟摘要

当前延迟结论：

1. `p95` 已进入当前阶段可接受区间
2. `p99` 仍存在长尾尖峰，但在 Phase 2.8 最后窗口内没有继续恶化
3. 当前仍不适合将这组结果直接外推为更高一级 rollout 的实施依据

## 4. 当前运行期风险清单

### A. 当前可接受风险

1. `p99` 仍存在长尾波动
2. 当前验证范围仅限已定义的局部启用范围
3. 当前运行基线尚未证明对更广 query 类型同样成立

这些风险当前可接受，因为：

1. 尚未进入更高一级 rollout 实施
2. 当前范围内仍保持可观测、可关闭、可回退
3. 没有破坏当前范围内的稳定运行

### B. 当前需要持续警惕的风险

1. 历史上出现过一次后端瞬时失败窗口
2. 该窗口曾导致 `dense_failed_count = 48`
3. 该窗口曾导致目标命中集临时整体归零

当前判断：

1. 该风险在 Phase 2.8 最后观察窗口内未复现
2. 但它仍然是 rollout 评估前必须继续关注的运行期风险

### C. 当前会阻止进入更高一级 rollout 评估的风险

满足以下任一情况，当前就不应进入更高一级 rollout 评估：

1. 后端异常窗口开始重复出现
2. `p99` 再次持续恶化
3. 命中集再次漂移
4. 新的误命中或漏命中边界样本开始出现
5. 全局关闭 / 局部关闭 / fail-open / trace 任何一项失去稳定性

## 5. 回滚与处置条件摘要

### 5.1 继续维持当前基线的前提

以下条件同时满足时，可以继续维持当前运行基线：

1. 当前启用范围不扩大
2. 当前规则不修改
3. 当前 provider 不变
4. 当前观测项持续可用
5. 当前关闭与回退逻辑持续有效

### 5.2 应优先全局关闭的条件

1. 后端异常窗口连续复现
2. 命中集再次整体归零
3. `fail_open_rate` 明显抬升并持续
4. `p99` 明显恶化并持续超出当前预算

### 5.3 应优先局部关闭的条件

1. 当前局部启用范围内重新出现误命中
2. 当前局部启用范围内重新出现漏命中边界样本
3. 当前局部范围的收益开始不稳定

## 6. 进入更高一级 Rollout 评估的前提

只有在以下条件同时满足时，才建议进入更高一级 rollout 评估：

1. 当前运行基线继续保持不变
2. 当前启用范围继续保持不变
3. 运行期核心指标继续稳定：
   - `rerank_hit_rate`
   - `fail_open_rate`
   - `latency p95`
   - `latency p99`
4. 未再次出现新的后端异常窗口，或异常窗口未形成重复趋势
5. 回滚条件明确且已验证可用
6. runbook、long-run 文档与 TODO 已一致
7. 双仓库可回滚基线已完成

### 18.1 本轮新增证据

本轮没有扩功能边界，而是围绕两个方向补证：

   继续补“多文件连续切换”和“运行稳定性证据”，但不把当前补证误写成已经完成高可信灰度测试。

### 18.2 结构化检索增强结果

本轮已确认的提升：

1. 资质等级 / 项目经理类问题  
   查询可显式收敛到“资格后审 / 资格审查 / 资信标”相关章节，不再完全散落在泛技术要求段落中。

2. 工程量清单 / 不平衡报价类问题  
   查询已能明显打到“第八章 工程量清单 / 最高投标限价”相关 chunk，章节命中质量优于此前普通 lexical 检索。

3. 合同专用条款相关商务参数  
   第二轮增强后，“付款节点 / 结算方式 / 缺陷责任期 / 质保金 / 误期赔偿”类查询已开始稳定打到“第三章 招标人对招标文件及合同范本的补充/修改”，不再持续被工程量清单或泛段落污染。

4. 多文件连续切换稳定性  
   本轮继续验证《附件十一：数字化交付标准》与《答疑补遗文件（一）》的往返切换，`applied_filters.document_id` 与返回结果中的 `document_id` 保持一致，未出现旧文件污染回归。

本轮仍未完全解决的问题：

1. 付款节点 / 结算方式  
   当前已经能稳定进入第三章合同补充条款，但 top-1 / top-2 仍可能落在“最终结清 / 竣工结算”附近，而不是最理想的支付条款 chunk。

2. 缺陷责任期 / 质保金  
   当前已能稳定进入第三章并命中“最终结清申请 / 质量保证金 / 缺陷责任期”相关内容，但仍未证明在不同大标书上都能稳定抽出最关键值。

3. 延期误期赔偿  
   当前已能稳定进入第三章合同补充条款，但返回内容仍可能混入“违约 / 最终结清 / 缺陷责任期”邻近段落，尚未形成足够干净的违约赔偿召回。

4. 总工期 / 关键节点  
   当前虽已补入“工期要求 / 计划开工日期 / 计划竣工日期 / 关键工期节点”等真实标题词，查询也能稳定进入 `schedule_scope`，但真实结果仍常回到“技术要求 / 进度延误措施 / 投标格式 / 工程量清单”邻近段落，尚未稳定命中最理想的前附表或合同条款 chunk。这一方向依然是当前最明显的短板。

这说明：

1. 章节感知检索已带来真实提升
2. 合同专用条款方向已经开始有效，但还没到“全面够用”
3. 前附表 / 工期节点方向仍需继续增强
4. 下一步仍需要更强的目录 / 标题感知和章节切分策略

### 18.3 高可信度灰度补证结果

本轮新增补证如下：

1. 多文件连续切换稳定性  
   《附件十一：数字化交付标准》与《答疑补遗文件（一）》之间的连续切换已验证稳定，`trace.sparse.backend = opensearch` 保持正常，未出现旧文件污染回归。

2. 大型标书标题切换  
   大型标书标题查询当前可返回正确文件内容，但在不带显式 `document_id` 时，document scope 仍可能停留在 `no_unique_document_match`，说明大文件标题归一化与自动收敛还不够稳。

3. 长窗口稳定性  
   当前没有新增高可信机器上的长窗口结果；Air 8GB 的既有结果仍只能作为受控验证依据，不能作为高可信最终裁决。

4. OpenSearch 超时 / `dense_failed_count` / `p95/p99` / fail-open

## 20. Phase 2.9c 兼容性回归修复结论

### 20.1 问题定性

本轮新增的 P0 问题不是 Windows 环境问题，也不是单个灰度脚本设计缺陷，而是一次真实的 service 兼容性回归：

1. `RetrievalService.search()` 新增 `_infer_document_scope()` 后，破坏了既有 `db=None` 兼容路径
2. `scripts/phase26_rerank_gray_validation.py` 使用：
   - `service = RetrievalService(db=None)`
3. `_infer_document_scope()` 之前会直接执行：
   - `self.db.query(Document)...`
4. 当 `db=None` 时，会直接报：
   - `AttributeError: 'NoneType' object has no attribute 'query'`

这属于：

1. 主线检索增强后
2. 未持续验证无 DB 评估脚本兼容性
3. 导致高可信度灰度补证入口被回归打断

### 20.2 修复内容

本轮已在 `RetrievalService` 层恢复 `db=None` 兼容性：

1. `_infer_document_scope()` 在无 DB 模式下安全返回：
   - `status = db_unavailable`
   - `reason = document_scope_requires_db`
2. `_database_fallback_search()` 在无 DB 模式下安全返回空结果

这样修的原因是：

1. repo 中存在多条 `RetrievalService(db=None)` 路径
2. 问题根因在 service 兼容性，而不在单个脚本
3. 只修单个脚本会继续把其他无 DB 评估路径暴露在同类回归风险下

### 20.3 影响面排查

#### A. 已确认受影响

1. `scripts/phase26_rerank_gray_validation.py`

原因：

1. 该脚本真实执行 `service.search()`
2. 会在 search 早期进入 `_infer_document_scope()`
3. 因而会直接触发 `NoneType.query` 回归

#### B. 潜在受影响

以下路径均使用 `RetrievalService(db=None)`，理论上都存在同类风险：

1. `scripts/phase23_rerank_spike.py`
2. `scripts/phase21_real_gate.py`
3. `tests/test_aliyun_reranker.py`
4. `tests/test_rerank_hook.py`
5. `tests/test_qdrant_dense_adapter.py`
6. `tests/test_dense_backend_integration.py`

原因：

1. 它们同样依赖无 DB 模式
2. 一旦执行 `search()` 并触发 document scope 推断，就可能踩到同类问题

#### C. 已验证不受影响

以下路径本轮已验证恢复到兼容状态：

1. `tests/test_retrieval_contract.py`
2. `tests/test_retrieval_fallback.py`
3. `tests/test_rerank_hook.py` 中本轮覆盖到的 fallback / policy 相关无 DB 路径
