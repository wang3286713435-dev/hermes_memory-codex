## 1. 路线图目标

本路线图用于指导 Hermes 从现有 Agent 升级为企业内核型 AI Agent。

路线图目标包括：

1. 先建立可靠的企业知识问答闭环。
2. 再增强知识治理、结构化事实和权限能力。
3. 最后扩展到更多企业系统和复杂 Agent 工作流。
4. 长期演进为支持多模态企业资料、会议记忆和经营辅助分析的企业级 Agent 中枢。

路线图不追求一次性完成所有能力，而是按风险和价值分阶段交付。

## 2. 阶段划分原则

1. 先底座，后智能。
2. 先检索，后生成。
3. 先引用，后自动化。
4. 先权限，后大规模接入。
5. 先评测，后持续优化。
6. 先核心资料，后全量系统。
7. 先关系型事实库，后知识图谱。
8. 先稳定记忆底座，后多模态和会议理解。
9. 先辅助分析，后高风险动作自动化。
10. 先资产目录和派生索引，后本体、知识图谱、空间索引和子 Agent 编排。

## 3. MVP 范围

MVP 的目标是验证 Hermes 是否具备企业长期记忆的基本闭环。

MVP 包含招标资料接入、公司核心文档接入、公司公众号文章接入、文档解析、文档清洗、结构化 chunking、metadata 提取、1024 维 embedding、现有向量数据库写入、OpenSearch BM25 索引、hybrid search、metadata filter、rerank、带引用问答、基础权限控制、文档版本记录、增量更新、审计日志和 100 条以上评测问题。

MVP 不包含 OA、CRM、ERP 全量接入、深度业务系统写操作、自动审批、完整知识图谱、多 Agent 自治工作流、Data Steward 产品化交付、TB 级 BIM 全量解析和私有大模型训练。

## 4. Phase 1：企业知识问答闭环

### 4.1 阶段目标

让 Hermes 能够基于企业文档进行可靠问答，并提供可追溯引用。

### 4.2 数据范围

1. 近 1-2 年招标资料。
2. 公司核心制度文件。
3. 典型项目方案。
4. 公司公众号历史文章。
5. 常用资质、案例、产品资料。

### 4.3 核心交付

1. 文档上传和批量导入。
2. PDF、Word、Excel、PowerPoint、HTML 基础解析。
3. 公众号文章接入。
4. 文档清洗。
5. 结构化 chunking。
6. 文档元数据提取。
7. 文档版本记录。
8. Embedding 服务。
9. 向量数据库写入。
10. OpenSearch BM25 写入。
11. Hybrid search。
12. Metadata filter。
13. Rerank。
14. 可引用问答。
15. 基础权限控制。
16. 增量更新基础能力。
17. 审计日志。
18. 基础评测集。

### 4.4 关键能力边界

Phase 1 只要求支持文档知识问答闭环，不要求完整结构化事实体系，不要求复杂知识图谱，不要求深度业务系统操作。

### 4.5 验收标准

1. 完成至少 3 类数据源接入。
2. 支持主流办公文档解析。
3. 支持 dense + BM25 hybrid search。
4. 支持 metadata filter。
5. 支持 rerank。
6. 回答带引用。
7. 文档版本可记录。
8. 文档更新可增量处理。
9. 权限测试无越权召回。
10. 建立 100 条以上评测问题。
11. 核心问题 Top5 命中率达到 80% 以上。
12. 引用准确率达到 85% 以上。

## 5. Phase 2：知识治理与结构化事实增强

### 5.1 阶段目标

提升知识质量、检索质量、结构化事实查询能力和权限治理能力。

### 5.2 数据范围

1. 历史投标文件。
2. 合同模板。
3. 项目交付文档。
4. 客户资料。
5. 会议纪要。
6. 市场素材。
7. 内部培训资料。
8. BIM / Data Steward 文件资产目录试点。

### 5.3 核心交付

1. 招标资料专项解析。
2. 表格结构增强解析。
3. 条款级 chunking。
4. 结构化事实抽取。
5. 客户实体库。
6. 项目实体库。
7. 资质实体库。
8. 案例实体库。
9. 版本对比。
10. 权限策略增强。
11. 用户反馈闭环。
12. 检索质量看板。
13. 自动评测流水线。
14. 知识管理员后台。
15. 人工校验机制。
16. 多模态企业资料摄取方案，包括 Excel、PowerPoint、图片、扫描件和会议音频的分阶段接入设计。
17. 会议记忆试点方案，包括转写、议题、决策、行动项和后续跟进事项的结构化沉淀。
18. BIM / Data Steward 文件资产治理试点方案，包括项目、楼栋、楼层、空间、专业、系统、版本、路径、hash、责任人和权限标签的目录治理。

### 5.4 关键能力边界

Phase 2 建设结构化事实层，但不要求完整知识图谱。事实抽取结果必须支持人工校验，不应默认完全可信。

### 5.5 验收标准

1. 核心实体可结构化查询。
2. 招标要求可提取为结构化字段。
3. 项目、客户、资质、案例支持关联查询。
4. 版本差异可查看。
5. 权限策略支持部门、项目、密级组合判断。
6. 用户反馈可进入评测闭环。
7. 评测集扩展至 300 条以上。
8. 核心问题 Top5 命中率达到 85% 以上。
9. 引用准确率达到 90% 以上。
10. 结构化事实关键字段人工抽检准确率达到 90% 以上。

## 6. Phase 3：企业系统扩展与 AI 中枢化

### 6.1 阶段目标

将 Hermes 扩展为连接更多企业系统的 AI 中枢。

### 6.2 数据范围

1. OA。
2. CRM。
3. ERP。
4. 项目管理系统。
5. 邮件系统。
6. 即时通讯记录。
7. 财务和经营数据。
8. 外部行业知识源。
9. Data Steward / BIM 建模文件资产目录和后续模型解析派生数据。

### 6.3 核心交付

1. 企业系统连接器。
2. 多部门知识空间。
3. 更完整结构化事实层。
4. 更细粒度字段权限。
5. 知识图谱试点。
6. 主数据治理集成。
7. 高可用部署。
8. 私有模型优化。
9. 复杂任务工作流。
10. 多 Agent 协作能力。
11. 企业会议记忆与经营辅助分析能力。
12. 多模态企业资料统一治理能力。
13. Data Steward 数据管家能力，包括模型文件资产目录、权限治理、版本追踪、派生元数据检索、建筑本体、知识图谱和空间索引后续入口。

### 6.4 关键能力边界

Phase 3 可逐步引入系统操作能力，但审批、财务、法务、投标最终提交等高风险动作必须保留人工确认。

多模态资料理解、会议记忆和经营辅助分析属于企业 AI 中枢化的重要方向，但不得绕过权限、审计、引用和人工确认边界。会议录音、财务经营数据和投标决策类内容必须具备可追溯来源和人工复核机制。

BIM / Data Steward 应先作为文件资产治理能力进入，不应直接承诺 TB 级原始模型全量解析、在线模型查看、碰撞检测、自动算量或自动设计审查。Hermes 可管理模型资产目录、派生文本 / 清单 / 属性索引、建筑本体关系和引用回链，但不得替代 BIM 协同平台、NAS、文件服务器、专业建模软件、楼宇自控系统或运维系统。

### 6.5 后置产品线：Data Steward

Data Steward 是 Hermes 企业内核能力的后置产品线，不抢当前 MVP Pilot 与 Phase 2.38d 的实现资源。推荐路线为：

1. Data Steward Planning：明确产品边界、客户场景、资产目录字段、本体范围和非目标。
2. Building Asset Catalog MVP：选择单栋楼 / 单项目目录，建立文件资产、版本、路径、hash、责任人、权限标签和派生索引。
3. Ontology / Knowledge Graph Pilot：抽取项目、楼栋、楼层、空间、构件、设备、传感器、系统、工单和责任人的实体关系，生成可审计三元组。
4. Spatial Index Pilot：为构件、设备、空间和点云派生边界盒、坐标、楼层、区域索引，支持后续空间查询入口。
5. Subagent Orchestration / Monitoring：规划 Data Steward Supervisor、Catalog Agent、Ontology Agent、Spatial Index Agent、Monitoring Agent 和 Review Agent；先监控与审阅，后自动化。

该产品线的商业化卖点应是“可审计数据治理 + 子 Agent 监控 + evidence answer”，不是承诺自动替代 BIM 平台、数字孪生平台、碰撞检查、自动算量或楼宇运维系统。

### 6.6 验收标准

1. 至少接入 2 个核心企业系统。
2. 支持跨系统检索和引用。
3. 支持更细粒度权限控制。
4. 支持系统级审计。
5. 支持高可用部署。
6. 支持关键任务人工确认。
7. 评测集扩展至 500 条以上。
8. 权限测试覆盖多系统数据。
9. 系统稳定支撑真实业务使用。

## 7. 阶段边界

### 7.1 Phase 1 与 Phase 2 边界

Phase 1 重点是文档问答闭环。Phase 2 重点是结构化事实、知识治理和质量提升。

### 7.2 Phase 2 与 Phase 3 边界

Phase 2 重点是内部知识治理。Phase 3 重点是企业系统扩展和 AI 中枢化。

### 7.3 MVP 与完整系统边界

MVP 只证明核心路径可用，不追求全量数据、全量权限模型、完整结构化事实和复杂自动化。

## 8. 阶段验收标准

### 8.1 MVP 验收

1. 三类数据源可接入。
2. 文档可解析。
3. Chunk 可索引。
4. Hybrid search 可用。
5. Rerank 可用。
6. 回答带引用。
7. 基础权限生效。
8. 可增量更新。
9. 有审计日志。
10. 有基础评测集。

### 8.2 Phase 1 验收

真实业务用户可使用，招标、制度、公众号、项目资料可检索，核心问题有稳定引用，无明显越权问题，检索质量可量化。

### 8.3 Phase 2 验收

结构化事实层、知识管理员后台、版本治理、反馈和评测闭环、增强权限策略均可用。

### 8.4 Phase 3 验收

跨系统知识查询、多系统权限治理、企业 AI 中枢能力和高风险动作人工确认机制可用。

## 9. 交付优先级

### 9.1 P0

文档接入、文档解析、结构化 chunking、embedding、向量检索、BM25、hybrid search、metadata filter、rerank、citation、permission control、versioning 和 evaluation。

### 9.2 P1

结构化事实层、知识管理员后台、版本对比、用户反馈闭环、评测看板和招标专项抽取。

### 9.3 P2

企业系统连接器、Data Steward、知识图谱、多 Agent 工作流、私有模型优化和主数据治理集成。

## 10. 推荐实施顺序

1. 先做企业知识底座。
2. 再做 hybrid search。
3. 再做带引用问答。
4. 再做权限和版本治理。
5. 再做增量更新和评测闭环。
6. 再做结构化事实层。
7. 最后扩展企业系统和复杂工作流。
8. 在记忆底座稳定后，再逐步引入 Excel / PowerPoint 结构化解析、会议音频记忆和经营辅助分析。

最小落地路径：

招标资料 + 公司核心文档 + 公众号文章 -> 文档解析与分块 -> hybrid search -> rerank -> 带引用回答 -> 权限与版本控制 -> 评测闭环。

长期扩展路径：

稳定文档记忆底座 -> 多文件会话作用域 -> Excel / PowerPoint 结构化摄取 -> 会议音频转写与会议记忆 -> 结构化事实联动 -> 企业经营辅助分析 -> Data Steward 资产目录 / 本体 / 图谱 / 空间索引试点 -> 人工确认边界内的工作流协助。
