## 1. 文档说明

本文档记录 Hermes 企业内核型 AI Agent 项目的关键技术选型与架构取舍。

每条 ADR 包含背景、决策、理由、影响和替代方案。

## 2. ADR-001：长期记忆系统不等于向量数据库

### 背景

项目已配置向量数据库，当前向量维度为 1024。但企业长期记忆系统不能只依赖向量数据库。

### 决策

Hermes 长期记忆系统必须包含文档知识层、结构化事实层、会话记忆层、检索层和权限治理层。向量数据库只作为检索层中的语义召回组件。

### 理由

向量数据库无法独立解决权限控制、文档版本管理、引用定位、结构化事实查询、审计追踪、增量更新、精确关键词检索、质量评测和数据治理。

### 影响

系统需要引入 PostgreSQL、OpenSearch、对象存储、权限服务、评测系统等组件。

### 替代方案

只使用向量数据库和大模型上下文。该方案实现简单，但无法满足企业级可信问答和权限治理要求，因此不采用。

## 3. ADR-002：采用 Hybrid Search 而非纯向量检索

### 背景

企业中文场景中，用户查询经常包含客户名、项目名、文件编号、招标编号、资质名称、金额、日期和条款原文。

### 决策

MVP 起即采用 hybrid search，包含 Dense Retrieval、BM25 / Sparse Retrieval、Metadata Filter、Rerank、Query Rewrite 和结构化事实召回。

### 理由

纯向量检索对精确字段召回不稳定，对中文短查询容易语义漂移，对专有名词和缩写不稳定，不能解决权限过滤、版本过滤和引用定位问题。

### 影响

系统需要维护向量索引和全文索引两套检索基础设施。

### 替代方案

只使用向量检索。该方案召回链路简单，但企业场景风险过高，因此不采用。

## 4. ADR-003：沿用现有 1024 维向量数据库作为 MVP 向量层

### 背景

当前系统已配置向量数据库，向量维度为 1024。

### 决策

MVP 阶段沿用现有向量数据库，不立即迁移向量基础设施。

### 理由

该决策可降低迁移风险，缩短 MVP 交付周期，保持与现有 Hermes 基础能力兼容。当前主要短板不在向量库本身，而在 hybrid search、metadata、rerank、citation、versioning、permission control 和 evaluation。

### 影响

需要为现有向量数据库补齐元数据映射和索引版本管理能力。

### 替代方案

迁移到 Qdrant 或 Milvus。该方案可作为后续规模化优化方向，MVP 不采用。

## 5. ADR-004：采用 OpenSearch 作为全文检索引擎

### 背景

系统需要 BM25、中文分词、关键词检索和元数据过滤能力。

### 决策

采用 OpenSearch 作为 MVP 全文检索引擎。

### 理由

OpenSearch 的 BM25 成熟，支持中文分词配置，支持丰富 metadata filter，企业私有化部署成熟，授权和成本相对可控。

### 影响

需要维护 OpenSearch 索引结构、中文分词配置和索引更新任务。

### 替代方案

Elasticsearch。能力相近，但 OpenSearch 在私有化和授权方面更适合作为默认推荐。

## 6. ADR-005：采用 PostgreSQL 作为元数据与结构化事实库

### 背景

系统需要保存文档、版本、chunk、权限、事实、会话记忆、审计和评测数据。

### 决策

采用 PostgreSQL 作为主关系数据库和结构化事实库。

### 理由

PostgreSQL 成熟稳定，支持复杂查询、事务和 JSONB，适合结构化事实和元数据管理，团队维护成本低。

### 影响

结构化事实层首先以关系模型实现，后续再评估图数据库。

### 替代方案

直接使用图数据库。该方案适合复杂关系分析，但 MVP 阶段复杂度较高，因此暂不采用。

## 7. ADR-006：采用 MinIO 作为对象存储

### 背景

系统需要保存原始文件、解析中间产物、OCR 结果和可能的导出文件。

### 决策

采用 MinIO 作为对象存储。

### 理由

MinIO S3 兼容，私有化部署成熟，运维成本低，适合企业内部文件存储。

### 影响

需要设计文件 bucket、路径规范、权限映射和生命周期策略。

### 替代方案

本地文件系统。该方案简单，但扩展性和可靠性不足，因此不作为默认方案。

## 8. ADR-007：采用 FastAPI 作为后端服务框架

### 背景

系统主要为 AI、文档处理、检索和企业内部 API 服务。

### 决策

采用 Python FastAPI。

### 理由

Python AI 生态成熟，文档解析、OCR、embedding、RAG 集成方便。FastAPI 性能和开发效率较好，自动生成 OpenAPI 文档，适合 MVP 快速交付。

### 影响

核心服务以 Python 为主，性能敏感组件可后续独立优化。

### 替代方案

Java Spring Boot。该方案企业成熟度高，但 AI 文档处理生态接入成本更高，MVP 不优先采用。

## 9. ADR-008：采用 Celery + Redis 作为异步任务系统

### 背景

文档摄取、解析、OCR、embedding、索引写入都是耗时异步任务。

### 决策

采用 Celery + Redis。

### 理由

Celery + Redis 成熟稳定，支持任务队列、失败重试，适合 Python 技术栈和批量文档任务。

### 影响

需要定义任务状态表、重试策略、死信处理和任务监控。

### 替代方案

RQ 或 Dramatiq。两者可用，但 Celery 生态和生产经验更成熟。

## 10. ADR-009：采用 PaddleOCR 作为 OCR 方案

### 背景

企业资料中存在扫描件、图片 PDF、盖章文件和截图。

### 决策

采用 PaddleOCR 作为 OCR 方案。

### 理由

PaddleOCR 中文识别能力较强，支持私有化部署，社区和实践案例较多，适合处理中文企业文档。

### 影响

OCR 结果需要质量标记，低质量内容应降低检索权重或进入人工复核。

### 替代方案

云 OCR API。识别效果可能更好，但敏感资料外发存在合规风险，因此不作为默认方案。

## 11. ADR-010：采用 bge-reranker 中文模型作为 MVP Rerank 方案

### 背景

Hybrid search 初步召回结果需要进行相关性重排。

### 决策

MVP 采用 bge-reranker 中文能力模型私有化部署。

### 理由

bge-reranker 中文相关性表现较好，可私有化部署，成本可控，与 RAG 场景匹配，并且后续可替换。

### 影响

Rerank 会增加延迟，需要控制候选数量和超时策略。

### 替代方案

使用大模型进行 rerank。效果可能更好，但成本和延迟较高，MVP 不优先采用。

## 12. ADR-011：采用 RBAC + ABAC 权限模型

### 背景

企业知识访问受角色、部门、项目、密级、文档类型和用户属性影响。

### 决策

采用 RBAC + ABAC 混合权限模型。

### 理由

RBAC 适合角色管理，ABAC 适合细粒度企业权限，支持部门、项目、密级组合判断，并可扩展到后续企业系统。

### 影响

权限判断必须贯穿摄取、索引、检索、prompt 构建、引用展示和审计。

### 替代方案

仅文档级 ACL。该方案简单，但无法满足复杂企业权限要求。

## 13. ADR-012：采用 Citation-first 回答策略

### 背景

企业场景中回答可信度和可追溯性比回答流畅度更重要。

### 决策

Hermes 默认采用 citation-first 策略。回答必须基于检索结果，引用文件、版本、页码和章节；无依据时说明不确定；冲突来源必须提示；权限不足不得泄露。

### 理由

该策略可降低幻觉风险，提高用户信任，支持审计追踪，适合投标、制度、合规等场景。

### 影响

Prompt 设计和回答格式必须围绕引用组织。

### 替代方案

直接生成自然语言答案。该方案体验流畅但企业风险高，不采用。

## 14. ADR-013：采用版本优先的知识治理策略

### 背景

企业文档存在多个版本，旧版本污染会导致错误回答。

### 决策

所有文档、chunk、索引和引用必须绑定版本信息。默认检索 latest + active 文档。

### 理由

该策略可避免旧知识污染回答，支持历史追溯、版本对比、制度和招标文件有效性判断。

### 影响

数据库、索引和引用结构都必须包含 version_id 和状态字段。

### 替代方案

只保留最新文件。该方案简单，但无法审计历史版本，也无法解释历史回答依据。

## 15. ADR-014：采用评测驱动的检索优化机制

### 背景

RAG 系统优化如果没有评测集，容易凭感觉调参。

### 决策

MVP 阶段即建设评测系统和评测集。评测指标包括 Recall@K、MRR、NDCG、Citation Accuracy、Answer Faithfulness、Version Accuracy 和 Permission Safety。

### 理由

评测机制可量化检索效果，防止改动引入回退，持续优化 chunking、hybrid 权重和 rerank，并支撑项目验收。

### 影响

研发流程中需要加入回归评测。

### 替代方案

上线后依赖用户反馈。该方案反馈慢且不可控，因此不采用。

## 16. 最终推荐技术栈

### 16.1 应用服务

最终推荐：Python FastAPI。

### 16.2 异步任务

最终推荐：Celery + Redis。

### 16.3 关系数据库

最终推荐：PostgreSQL。

### 16.4 向量数据库

最终推荐：沿用现有 1024 维向量数据库。

### 16.5 全文检索

最终推荐：OpenSearch。

### 16.6 对象存储

最终推荐：MinIO。

### 16.7 OCR

最终推荐：PaddleOCR。

### 16.8 文档解析

最终推荐：PyMuPDF、pdfplumber、python-docx、openpyxl、python-pptx、BeautifulSoup 和 LibreOffice headless。

### 16.9 Rerank

最终推荐：bge-reranker 中文模型私有化部署。

### 16.10 可观测性

最终推荐：OpenTelemetry、Prometheus、Grafana 和 Loki。

### 16.11 部署形态

MVP 推荐模块化单体 + 独立基础组件。后续当数据量、并发和团队规模增长后，再拆分为独立微服务。
