# Phase 2.1-Qdrant Dense Retrieval Backend

## 1. Scope

Phase 2.1-Qdrant replaces the temporary dense `/search` backend route with Qdrant as the primary vector store backend. This phase keeps the existing retrieval request/filter/result contract stable and uses adapter normalization to map Qdrant results into `SearchResult`.

This phase does not implement rerank, facts joins, OCR, multi-agent workflows, complex permission policy, large ingestion refactors, or ChromaDB continuation.

## 2. Decision

The project adopts Qdrant as the default dense retrieval backend:

- `VECTOR_STORE_PROVIDER=qdrant`
- `QDRANT_URL=http://qdrant:6333` in Docker development
- `QDRANT_COLLECTION=hermes_chunks`
- `QDRANT_VECTOR_SIZE=1024`

ChromaDB is not continued as the project main route. The reason is operational clarity: Qdrant provides mature collection management, payload filtering, top-k vector search, and production deployment patterns, reducing custom dense retrieval code.

## 4. Collection Design

Collection:

- name: `QDRANT_COLLECTION`, default `hermes_chunks`
- vector size: `QDRANT_VECTOR_SIZE`, default `1024`
- distance: `Cosine`

Point id:

- `chunk_id`

Vector:

- 1024-dimensional embedding generated by Aliyun `text-embedding-v3`, or supplied as `query_vector` during tests/manual validation.

## 5. Payload Mapping

Each point payload should include:

- `chunk_id`
- `document_id`
- `version_id`
- `text`
- `source_type`
- `document_type`
- `is_latest`
- `source_name`
- `source_uri`
- `version_name`
- `heading_path`
- `section_path`
- `page_start`
- `page_end`
- `metadata`

The adapter maps Qdrant result payloads to the existing `SearchResult` contract and sets:

- `retrieval_sources=["dense"]`
- `scores={"dense": score}`

## 6. Filter Mapping

Supported filters in this phase:

- `source_type`
- `document_id`
- `document_type`
- `is_latest`

They are mapped to Qdrant `must` match clauses:

```json
{
  "must": [
    {"key": "source_type", "match": {"value": "upload"}},
    {"key": "document_id", "match": {"value": "doc-1"}}
  ]
}
```

Unsupported filters continue to be reported through `ignored_filters` before reaching the adapter.

## 7. Embedding Path

Query embedding is handled by `AliyunTextEmbeddingV3`:

- model: `text-embedding-v3`
- dimension: `1024`
- API key: `ALIYUN_EMBEDDING_API_KEY`

If `query_vector` is supplied in `SearchRequest`, the adapter uses it directly. If no vector is supplied and the Aliyun key is missing, dense retrieval returns `unavailable` and hybrid retrieval fails open to sparse.
